## 📁 Project Structure

```
lib/
├── main.dart              # Entry point — routes users using StreamBuilder auth listener
├── firebase_options.dart  # Firebase settings generated by FlutterFire CLI
├── core/
│   └── services/          # Services (Firebase, Location, Map, Routing)
├── models/                # Data models (PlaceModel)
├── providers/             # State management (PlaceProvider)
├── screens/               # Screen views
│   ├── auth/              # login_screen.dart (Firebase Login/Signup)
│   ├── home/              # home_screen.dart (Workshop list, search & filters, logout)
│   ├── detail/            # place_detail_screen.dart (Detailed workshop info & route link)
│   ├── map/               # map_screen.dart (Google Map path plotting)
│   └── favorite/          # favorite_screen.dart (Saved favorites screen)
└── widgets/               # Reusable UI widgets (PlaceCard, FilterSheet)
```

**Current state:** The project is a fully functional Workshop Directory application. It is integrated with Firebase Console (project `bengkel-directory`) for Firestore storage and Firebase Authentication. The login/register UI has been modernized with custom text field validations, and the database photo URL fetching includes a real-time fallback mechanism that automatically queries Flickr search images via LoremFlickr based on workshop names and categories.

## 🔀 Routes / Navigation

Current routing is set dynamically in `main.dart` using a `StreamBuilder` on `FirebaseAuth.instance.authStateChanges()`:

| Route / State | Target Screen | Guard |
|------------|------|--------|
| Unauthenticated / Sign Out | `LoginScreen` | Redirects here when logged out |
| Authenticated | `HomeScreen` | Redirects here when logged in |
| Navigation from Home | `FavoriteScreen` | Push Route |
| Navigation from Home | `MapScreen` | Push Route |
| Navigation from Home / Card | `PlaceDetailScreen` | Push Route |

## 🧠 Instructions for AI Assistants

### Before making any change:
1. **Read this file completely**
2. **Read `DEVELOPMENT_GUIDELINES.md`** for coding conventions
3. **Check the Features table** to understand what exists
4. **Check the Dependencies table** before adding packages

### After making any change:
1. **Update the Features table** if you added/modified a screen
2. **Update the Dependencies table** if you added/removed a package
3. **Update the Data Models section** if you created/changed a model
4. **Update the API Endpoints section** if you added API calls
5. **Update the Routes section** if you added navigation routes
6. **Update the Changelog** with a one-line summary of what you did
7. **Update "Current state"** description if the project status changed

### Code placement rules:
- New screens → `lib/screens/`
- New reusable widgets → `lib/widgets/`
- New data models → `lib/models/`
- New API/service logic → `lib/core/services/`
- New state management → `lib/providers/`
- Constants/config → `lib/config/`
- Helper functions → `lib/utils/`
- **NEVER** put everything in `main.dart`

### Style rules:
- File names: `snake_case.dart`
- Class names: `PascalCase`
- Variables/functions: `camelCase`
- Constants: `SCREAMING_SNAKE_CASE`
- Max 50 lines in a `build()` method — extract sub-widgets
- Group imports: Dart SDK → Flutter → Third-party → Project
